Vite 的依赖预构建是**开发环境**下的一项核心优化机制：当项目首次启动时，它会基于 `optimizeDeps` 配置，使用 `esbuild` 预先处理项目中用到的第三方依赖（如 react、lodash-es 等），将其转换为 ESM 格式并缓存到本地，从而提升冷启动速度和首屏渲染性能，也让后续开发启动更快、页面加载更流畅。

> 只在开发环境下进行预构建，生产环境则直接使用原始依赖进行构建。

## 为什么需要“依赖预构建”？

这项优化主要解决了两个在开发环境中可能遇到的瓶颈：

1. **兼容性转换**：开发服务器要求所有代码均为原生 ES 模块（ESM），因此Vite需要先将`node_modules`中常见的 `CommonJS` 或 `UMD` 格式的依赖，预先转换为 `ESM` 格式，确保浏览器能直接加载运行。
2. **请求合并与性能优化**：某些依赖（如`lodash-es`）内部可能包含数百个互相引用的文件。浏览器在导入一个函数时，可能需要发出数百个 HTTP 请求，造成网络阻塞。预构建能将这些内部模块合并成一个文件，极大地减少了请求数量，提升页面加载速度。

## 工作原理

预构建流程由Vite自动管理，主要在`vite`命令启动开发服务器时执行。

1. **依赖发现**：Vite会扫描项目所有源文件，查找如 `import React from 'react'` 这样的“裸导入”，并将其作为预构建的入口点。
2. **执行构建**：启动后，Vite会调用**esbuild**（在Vite 6中切换为**Rolldown**）来执行高效的打包。构建后的文件会输出到`node_modules/.vite/deps`目录下。
3. **结果缓存**：为了提高二次启动速度，Vite会对预构建结果进行强缓存。只要依赖版本和内容未变，后续启动时就会直接使用缓存。
4. **动态更新**：开发过程中，如果引入了新的、未预构建的依赖，Vite会自动在后台重新构建并刷新页面。

### 预构建时机

Vite的依赖预构建执行时机贯穿了开发服务器的生命周期，主要包括**项目首次启动时、开发过程中动态发现新依赖时、以及依赖或配置发生变更时**这几个关键节点。

具体可以从以下几个方面来理解：

| 时机 | 触发条件 | 行为表现 |
| :--- | :--- | :--- |
| **首次启动时** | 执行`vite`命令，且`node_modules/.vite`目录下**无缓存** | **同步执行**：Vite 在启动服务器前进行依赖扫描和预构建，完成后才会启动服务器。你会在控制台看到类似“Pre-bundling them...”的日志。 |
| **开发过程中动态新增** | 代码中引入了一个**未被预构建**的新依赖（例如懒加载的路由组件） | **异步动态执行**：Vite会在请求到达时自动重新构建该依赖并刷新页面，但这可能会导致短暂的页面重载。可通过`optimizeDeps.include`手动预声明这类依赖来避免。 |
| **依赖或配置变更后重启** | `package.json`、`lockfile`、`vite.config.js`等**关键文件发生变化** | Vite会根据文件内容的哈希值判断缓存是否失效。若失效，则下次启动时会**自动重新执行预构建**。 |
| **手动强制触发** | 执行`vite --force`，或删除`node_modules/.vite`目录 | **强制执行**：完全忽略现有缓存，对所有依赖进行全新预构建，可用于解决缓存导致的异常问题。 |

## Monorepo 和链接依赖

在 Monorepo 项目中，可能会有一些依赖是通过 `npm link` 或 `yarn link` 方式链接的本地包。

`vite` 将链接的依赖视为源码。它不会尝试打包被链接的依赖，而是会分析被链接依赖的依赖列表。

- **链接包本身**：会像项目里的普通 `.js` 或 `.vue` 文件一样，被当作源码，未经打包就直接传递给浏览器。
- **链接包的依赖项**：会被分析出来，像任何普通外部依赖一样，被收集并进行预构建（Pre-Bundling） 后传递给浏览器。

这样修改后热更新（HMR）会更快，因为没有重新打包的开销。

然而，这需要被链接的依赖被导出为 ESM 格式。如果被链接的依赖不是 ESM 格式（例如 CommonJS 格式），那么你可以在配置里将此依赖添加到 `optimizeDeps.include` 中强制预构建这个链接的依赖。

```js
export default defineConfig({
  optimizeDeps: {
    include: ['linked-dep'],
  },
})
```

**修改链接依赖后 `--force` 重启**：因为默认不预构建，且 `node_modules/.vite` 有缓存，修改了链接依赖后浏览器不会自动更新。必须用 `--force` 命令行选项强制清除缓存并重启服务

## 自定义配置

你可以在项目根目录的 `vite.config.js` 文件中，通过 `optimizeDeps` 字段来配置预构建行为。

| 配置项 | 作用 | 使用场景示例 |
| :--- | :--- | :--- |
| **`include`** | 强制指定某些依赖必须进行预构建。 | - 项目工作在Monorepo下，包含未预构建的链接包。<br>- 插件动态导入的依赖，扫描时无法被发现。<br>- 强制优化一个内部包含大量内部模块的ESM库。 |
| **`exclude`** | 强制将某些依赖排除在预构建之外。 | - 依赖本身很小且已经是有效的ESM格式，可以直接被浏览器加载。 |
| **`entries`** | 指定用于发现依赖的入口文件，当默认的入口检测不准确时使用。 | 项目是一个多页面应用（MPA），默认的 `index.html` 无法涵盖所有入口。 |
| **`force`** | 设置为`true`时，会强制Vite重新执行预构建，忽略现有缓存。 | - 依赖包更新后，修改未生效。<br>- 怀疑当前缓存有问题，需要强制刷新。 |

### 配置示例

```javascript
// vite.config.js
import { defineConfig } from 'vite'

export default defineConfig({
  optimizeDeps: {
    // 强制预构建这些依赖
    include: ['linked-dep', 'lodash-es', 'some-slow-lib'],
    // 排除某些依赖（适用于已经是高效ESM的包）
    exclude: ['a-small-esm-lib'],
    // 指定自定义入口
    entries: ['src/main.js', 'src/pages/**/*.js']
  }
})
```

## 缓存

### 文件系统缓存

Vite 将预构建的依赖项缓存到 `node_modules/.vite` 中。它会基于以下几个来源来决定是否需要重新运行预构建步骤：

- 包管理器的锁文件内容，例如 `package-lock.json`，`yarn.lock`，`pnpm-lock.yaml`，或者 `bun.lock`；
- 补丁文件夹的修改时间；
- `vite.config.js` 中的相关字段；
- `NODE_ENV` 的值。

只有在上述其中一项发生更改时，才需要重新运行预构建。

如果出于某些原因你想要强制 Vite 重新构建依赖项，你可以在启动开发服务器时指定 `--force` 选项，或手动删除 `node_modules/.vite` 缓存目录。

### 浏览器缓存

已预构建的依赖请求使用 HTTP 头 `max-age=31536000, immutable` 进行强缓存，以提高开发期间页面重新加载的性能。一旦被缓存，这些请求将永远不会再次访问开发服务器。

如果安装了不同版本的依赖项（这反映在包管理器的 lockfile 中），则会通过附加版本查询自动失效。

如果你想通过本地编辑来调试依赖项，您可以：

1. 通过浏览器开发工具的 Network 选项卡暂时禁用缓存。**注意：这会导致每次页面重新加载时都绕过预构建缓存，可能会显著降低性能。**
2. 重启 Vite 开发服务器指定 `--force` 选项，来重新构建依赖项。
3. 重新载入页面。
