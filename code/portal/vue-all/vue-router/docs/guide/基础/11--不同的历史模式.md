在创建路由器实例时，history 配置允许我们在不同的历史模式中进行选择。

## Hash 模式

`hash` 模式是用 `createWebHashHistory()` 创建的：

```js
import { createRouter, createWebHashHistory } from 'vue-router'

const router = createRouter({
  history: createWebHashHistory(),
  routes: [
    //...
  ],
})
```

它在内部传递的实际 `URL` 之前使用了一个哈希字符（`#`）。由于这部分 `URL` 从未被发送到服务器，所以它不需要在服务器层面上进行任何特殊处理。不过，它在 `SEO` 中确实有不好的影响。如果你担心这个问题，可以使用 `HTML5` 模式。

### 原理

`hash` 模式的底层原理可以概括成一句话：**把“路由状态”存在 URL 的 fragment，也就是 `#` 后面，然后监听这个 fragment 的变化，在前端自己完成页面切换。**

先看一个例子：

```text
https://example.com/#/article/42
```

这里 URL 分两部分：

- `https://example.com/`：真正会参与 HTTP 请求的部分
- `#/article/42`：`hash`，也叫 fragment，只在浏览器端使用

关键点是：**浏览器发请求时不会把 `#` 后面的内容带给服务器**。所以不管你访问的是 `/#/home` 还是 `/#/about`，服务端看到的通常都只是 `/`，然后返回同一个前端入口页面。

**1. 为什么改了 `#` 不会整页刷新**

因为 `hash` 最初设计出来不是给前端路由用的，而是给页面内锚点定位用的，比如：

```html
<a href="#section3">跳到第 3 节</a>
```

当 URL 的 `hash` 变化时，浏览器默认行为是：

- 更新地址栏
- 记录一条历史记录
- 尝试跳转到页面内对应锚点
- 触发 `hashchange` 事件

但它**不会重新发起页面请求**。这正好被 SPA 拿来复用成“伪路径切换”。

**2. 前端路由是怎么接管它的**

像 `vue-router` 这样的路由库会做三件事：

1. 启动时读取当前 URL 的 `hash`
2. 监听 `hashchange` 事件
3. 每次变化后重新做一次“路径匹配 -> 渲染组件”

可以把它理解成这样一个简化流程：

```js
function getCurrentPath() {
  return location.hash.slice(1) || '/'
}

function render() {
  const path = getCurrentPath()
  const component = matchRoute(path)
  mount(component)
}

window.addEventListener('hashchange', render)
window.addEventListener('load', render)
```

例如当前地址从：

```text
/#/users
```

变成：

```text
/#/users/1
```

浏览器不会刷新页面，但会触发 `hashchange`，然后路由系统重新匹配 `/users/1`，把对应组件显示出来。

**3. 它为什么天然不容易 404**

这是 `hash` 模式最实际的好处。

因为服务端根本看不到 `#` 后面的路由信息，所以用户刷新：

```text
https://example.com/#/profile
```

服务端收到的仍然只是：

```text
GET /
```

只要服务器能返回你的 `index.html`，前端应用就能启动，然后再从 `location.hash` 里拿到 `/profile`，恢复当前页面。

这和 `history` 模式不同。`history` 模式下刷新 `/profile`，服务端真的会收到 `GET /profile`，如果没有配置重写规则，就会直接 404。

**4. 浏览器历史记录为什么也能工作**

修改 `hash` 会进入浏览器历史栈，所以：

- 点击链接切换 `#/a` -> `#/b`
- 浏览器会记录历史
- 点后退时，URL 会回到 `#/a`
- 再次触发 `hashchange`
- 前端重新渲染 `a` 页面

所以用户会觉得“前进后退都正常”，虽然本质上没有发生传统多页跳转。

**5. 一句话总结它的执行链路**

用户操作链接  
$\rightarrow$ 修改 `location.hash`  
$\rightarrow$ 浏览器触发 `hashchange`  
$\rightarrow$ `vue-router` 读取新 hash  
$\rightarrow$ 路由匹配  
$\rightarrow$ 渲染对应组件  
$\rightarrow$ 页面看起来像“切换了路由”，但实际上没有整页刷新

## HTML5 模式（history 模式）

用 `createWebHistory()` 创建 `HTML5` 模式（使用浏览器的 HTML5 `History API` 来管理路由），**推荐使用这个模式**：

```js
import { createRouter, createWebHistory } from 'vue-router'

const router = createRouter({
  history: createWebHistory(),
  routes: [
    //...
  ],
})
```

当使用这种历史模式时，`URL` 会看起来很 "正常"，例如 <https://example.com/user/id>。

不过，问题来了。由于我们的应用是一个单页的客户端应用，直接访问某个深层路径时，服务器可能返回 `404`。

原因是：

- 假设我们要访问 `/about` 路由
- 在前端跳转 `/about` 时，Vue Router 能接管
- 但你直接刷新 `/about`，浏览器会向服务器请求 `/about`
- 如果服务器不知道这是前端路由，就会认为这个路径不存在

所以 `history` 模式必须配合 **服务端回退到** `index.html`。

服务器需要做什么，原则很简单：

- 如果请求的是静态资源，就正常返回
- 如果不是已知文件，就返回 `index.html`
- 然后由 Vue Router 在前端接管路由解析
- 比如常见思路是“所有非静态资源请求都重写到 `index.html`”。

### 原理

Vue Router 在 history 模式下主要依赖这些浏览器能力：

- `history.pushState()`：跳转到新路由，但不刷新页面
- `history.replaceState()`：替换当前路由记录
- `popstate` 事件：监听浏览器前进、后退

所以当你在单页应用里切换页面时，URL 会变化，但页面不会整页刷新，Vue 只会切换对应组件。

## hash 模式导航路由详细说明

在 `hash` 模式里，前端路由上的 `/a`、`/b`，实际地址通常是：

```text
http://example.com/#/a
http://example.com/#/b
```

### 1. 直接在浏览器地址栏输入 `/a`

假设你直接打开：

```text
http://example.com/#/a
```

浏览器发给服务器的请求其实是：

```http
GET / HTTP/1.1
Host: example.com
```

注意，不会把 `#/a` 发过去。因为 `#` 后面的内容是 fragment，只存在浏览器本地。

服务器通常返回：

```http
HTTP/1.1 200 OK
Content-Type: text/html
```

返回内容一般是前端入口页面，比如 `index.html`。  
然后浏览器解析这个 HTML，再继续请求里面引用的静态资源，比如：

```http
GET /assets/app.js
GET /assets/app.css
GET /assets/vendor.js
```

服务器再分别返回这些 JS、CSS 文件。

当前端 JS 跑起来后，它会读取当前地址里的 `location.hash`，发现是：

```text
#/a
```

`/a` 组件的内容在 `app.js` 里面，然后 Vue Router 会将这个组件的内容渲染到 `index.html` 的 `<div id="app"></div>` 里。

如果 `/a` 组件是懒加载的，那么前端会额外发一个请求加载 `/a` 组件的 JS chunk，比如：

```http
GET /assets/a.js
```

### 2. 已经打开了 SPA，再从 `/a` 切到 `/b`

如果你是在单页应用里点击链接，从：

```text
http://example.com/#/a
```

切到：

```text
http://example.com/#/b
```

这一步通常：

- **不会重新请求服务器**
- **不会重新请求 `index.html`**
- **不会整页刷新**

浏览器只是本地修改了 URL 的 hash 部分，然后触发 `hashchange` 事件。  
`vue-router` 监听到这个事件后，在前端切换到 `/b` 对应的组件。

所以这一步对服务器来说，通常是：

```text
没有新的页面请求
```

服务器也就：

```text
没有新的页面响应
```

**例外情况**

如果 `/b` 页面组件用了懒加载，例如：

```js
const B = () => import('./views/B.vue')
```

那切换到 `/b` 时，浏览器可能会额外请求一个新的 JS chunk，比如：

```http
GET /assets/B.xxx.js
```

服务器返回这个 chunk 文件后，前端再渲染 `/b` 页面。  
这不是“路由请求”，而是“静态资源请求”。

### 3. 总结

**可以把整个过程理解成这样**

访问 `/#/a`：

1. 浏览器请求 `/`
2. 服务器返回 `index.html`
3. 浏览器再请求 JS/CSS
4. 前端读取 `#/a`
5. 渲染 `/a`

在页面里从 `/#/a` 切到 `/#/b`：

1. 浏览器只修改 hash
2. 不发页面请求给服务器
3. 前端监听到变化
4. 渲染 `/b`
5. 如果 `/b` 懒加载，额外请求对应 JS 文件

## history 模式导航路由详细说明

在 Vue Router 的 `history` 模式下，导航路由要分两种情况看：**前端应用已经加载完成后的路由切换**，和 **浏览器直接访问某个 URL**。

下面以导航到 `/a` 和 `/b` 这两个路由为例，来说明它们的区别：

### 1. 直接在浏览器地址栏输入 `/a`

这时浏览器会真的向服务器发请求：

```http
GET /a
```

如果服务器已经按 SPA 配好了回退规则，它通常会：

- 返回 `index.html`

然后浏览器再继续请求：

```http
GET /assets/index.js
GET /assets/index.css
```

前端 JS 启动后：

- Vue Router 读取当前地址 `/a`
- 匹配到 `/a` 对应组件
  - 路由不是懒加载：`/a` 组件已经在 `index.js` 里了，直接渲染 `/a` 页面
  - 路由是懒加载：前端会发一个请求加载 `/a` 组件的 JS 文件，加载完成后再渲染 `/a` 页面
- Vue Router 渲染 `/a` 页面，将 `/a` 组件的内容放到 `index.html` 的 `<div id="app"></div>` 里

访问 `/b` 也是一样：

```http
GET /b
```

服务器通常仍然返回：

- `index.html`

之后前端再加载资源并渲染 `/b` 对应组件。

所以在 `history` 模式里，**直接访问 `/a` 和 `/b`，服务器返回的常常是同一个 `index.html`**，它们的区别在 Vue Router 根据路径动态渲染不同组件的内容。

### 2. 已经打开了 SPA，再从 `/a` 切到 `/b`

如果当前页面已经把 Vue 应用加载起来了，比如你现在就在站点里：

- 当前地址是 `/a`
- 你点击了一个 `router-link` 跳到 `/b`

这时通常：

- 浏览器**不会因为路由切换本身再发一个 HTML 页面请求**
- Vue Router 调用 `history.pushState()` 修改地址栏为 `/b`
- 前端根据 `/b` 匹配到对应组件，然后渲染页面

也就是说，对服务器来说，**单纯这次 `/a -> /b` 的前端路由跳转，通常没有新的“页面请求”**。

服务器之前真正收到过的请求，通常是首次进入应用时这些：

- `GET /` 或 `GET /a`
- `GET /assets/index.js`
- `GET /assets/index.css`
- 以及页面里需要的图片、字体、接口请求等

但是如果 `/b` 是一个懒加载组件，那么前端会发一个请求加载 `/b` 组件的 JS 文件：

```http
GET /assets/b.js
```

### 3. 为什么在浏览器输入不同的路由，服务器返回的是同一个 `index.html`

因为 Vue 单页应用一般只有一个入口页。  
服务器并不知道 `/a` 和 `/b` 分别对应哪个 Vue 组件，它只负责把应用壳子交给浏览器：

```html
<html>
  <body>
    <div id="app"></div>
    <script src="/assets/index.js"></script>
  </body>
</html>
```

然后由前端代码决定：

- 当前是 `/a` 就渲染 A 页面
- 当前是 `/b` 就渲染 B 页面

### 4. 如果服务器没配好路由重写会发生什么

如果你直接访问 `/a`，浏览器会发：

```http
GET /a
```

但服务器如果把它当成真实文件路径处理，而又找不到 `/a`，就会返回：

```http
404 Not Found
```

这就是为什么 `history` 模式必须配置“未知路径回退到 `index.html`”。

## history 模式与 hash 模式的区别

`hash` 模式：

- URL 带 `#`，可读性稍差
  - 例子：`https://example.com/#/user/1`
- 不需要服务端额外配置，即不用配置服务器回退到 `index.html`
- 刷新深层页面通常不会 404
- 当在浏览器输入 `url` 访问时，发送到服务器的 `url` 始终是 `/`，不包含路由信息（hash 内容）。始终返回 `index.html` 后，前端根据 `location.hash` 来渲染对应页面

`history` 模式：

- URL 干净
  - 例子：`https://example.com/user/1`
- 需要服务器支持回退，即需要配置服务器回退到 `index.html`
- 更适合正式站点
- 当在浏览器输入 `url` 访问时，发送到服务器的 `url` 包含路由信息，例如 `/a`、`/b`，需要服务器正确处理这些路径。始终返回 `index.html` 后，前端根据 `location.pathname` 来渲染对应页面

## 服务器配置示例

注意：以下示例假定你正在从根目录提供服务。如果你部署到子目录，你应该使用 `Vue CLI` 的 `publicPath` 配置和相关的路由器的 `base` 属性。你还需要调整下面的例子，以使用子目录而不是根目录（例如，将`RewriteBase/` 替换为 `RewriteBase/name-of-your-subfolder/`）。

### Apache

```txt
<IfModule mod_negotiation.c>
  Options -MultiViews
</IfModule>

<IfModule mod_rewrite.c>
  RewriteEngine On
  RewriteBase /
  RewriteRule ^index\.html$ - [L]
  RewriteCond %{REQUEST_FILENAME} !-f
  RewriteCond %{REQUEST_FILENAME} !-d
  RewriteRule . /index.html [L]
</IfModule>
```

### nginx

```nginx
location / {
  try_files $uri $uri/ /index.html;
}
```

### 原生 Node.js

```js
const http = require('http')
const fs = require('fs')
const httpPort = 80

http
  .createServer((req, res) => {
    fs.readFile('index.html', 'utf-8', (err, content) => {
      if (err) {
        console.log('We cannot open "index.html" file.')
      }

      res.writeHead(200, {
        'Content-Type': 'text/html; charset=utf-8',
      })

      res.end(content)
    })
  })
  .listen(httpPort, () => {
    console.log('Server listening on: http://localhost:%s', httpPort)
  })
```
