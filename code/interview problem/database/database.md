## 1. 数据库索引是什么，如果每个 column 都加上会怎么样

数据库索引是一种用于提高数据检索速度的数据结构，类似于书籍的目录。它通过在表的某些列上建立排序的存储结构，让数据库可以快速定位到符合条件的行，而无需全表扫描。索引通常基于B树、哈希表等实现，但会占用额外存储空间，并在数据增删改时产生维护开销。

如果为每一列都单独建立索引，可能会带来以下后果：

1. **查询性能提升有限**  
   - 对于精确匹配或范围查询的列，索引确实能加速，但查询可能只涉及部分列，多余索引无法发挥作用。  
   - 如果多个索引存在，数据库优化器需要从中选择最佳方案，有时反而会增加规划时间。
   - 单列索引无法替代复合索引。例如，`WHERE a=1 AND b=2` 通常只能用上 `(a)` 或 `(b)` 其中一个索引，另一个条件需要回表过滤。而 `(a,b)` 复合索引效率更高。

2. **写操作性能严重下降**  
   - 每次执行 `INSERT`、`UPDATE` 或 `DELETE` 操作时，所有索引都需要同步更新，导致写操作的延迟明显增加，在高并发场景下可能成为瓶颈。
   - 每插入一行或更新任一列，数据库需要维护 N 个索引结构（N = 列数）。

3. **存储空间大幅膨胀**  
   - 每个索引都是一份独立的数据结构，会占用大量磁盘空间。当表数据量巨大且列数众多时，索引占用的空间可能远超数据本身。

4. **维护成本高**  
   - 索引需要定期重建、整理碎片，过多索引会加大数据库的维护负担，备份和恢复的时间也会延长。

5. **索引效果不佳**  
   - 对于低选择性的列（如性别、状态等仅有少量值的列），索引的过滤效果差，却仍要承担维护开销，得不偿失。

因此，在实际应用中应只为经常用于查询条件（WHERE、JOIN、ORDER BY等）的列创建索引，并综合考虑读写比例、数据分布等因素，避免过度索引。
