## 1. 中兴那个系统是怎么做权限管理的。不是用一个 yml 文件配置吗，怎么和数据库相关的

权限数据是存储在数据库里的，yml 文件只是用来配置权限的层级关系和一些元信息的。系统启动时会读取 yml 文件，解析出权限树结构，然后根据数据库中的权限数据来判断用户是否有访问某个资源的权限。这样做的好处是权限数据可以动态修改，不需要重启系统，同时也能保证权限结构的清晰和可维护性。

## 2. 前端你是怎么获取这个权限数据的

前端在用户登录成功后会从后端接口获取用户的权限数据，通常是一个包含用户权限列表或者权限树的 JSON 对象。这个数据会被保存在前端的状态管理工具（比如 Vuex）或者 `localStorage` 中，以便在用户访问不同页面时进行权限校验和界面展示的控制。每次用户访问需要权限的资源时，前端会根据保存的权限数据来判断是否允许访问或者显示相应的功能按钮。

## 3. 说一下平时是怎么用 ai 工具的

> “我平时在工作中会高频使用 AI 工具，主要是 GitHub Copilot。
>
> 具体分几个场景：
>
> 1. **代码生成** – 比如写一个深拷贝函数或者复杂的递归菜单，我只要写清楚注释，Copilot 就能补全大部分，我再根据项目规范调整。
> 2. **调试和解释代码** – 遇到看不懂的报错或者历史遗留代码，我会把关键片段和错误信息贴给 AI，让它帮分析原因，经常能快速定位到变量作用域或异步顺序的问题。
> 3. **重构优化** – 比如一个 useEffect 依赖太多、逻辑太乱，我会问 AI 怎么拆分成多个独立 hook，它给出方案后我再判断是否适合当前场景。
> 4. **生成 mock 数据和测试用例** – 给定一个 TypeScript 接口，让 AI 自动生成几组合理的假数据，省去手写的时间。
>
> AI给的代码我必须自己跑通、理解、并能向同事解释。有一次 AI 给我写了一个递归遍历树形菜单，看起来没问题，但没考虑深层爆栈的风险，我后来加上了迭代方案和深度限制。

## 4. 有准备一个 demo 能看一下吗

**场景一：你真的有准备 demo（最优情况）**

如果你在 GitHub、个人网站或个人项目里有可展示的 demo，这是最加分的情况。

> **回答话术：**
> “有的，我平时学习或做技术调研的时候，会习惯写一些小 demo 来验证想法。比如我之前研究 **虚拟列表 / WebSocket 实时通信 / 组件库按需加载** 的时候，专门写了一个 demo 放在了 GitHub 上。
>
> 屏幕共享的话，我可以给你演示一下。这个 demo 主要实现了 [核心功能]，比如点击这里的按钮，可以看到性能从 XX 提升到了 YY。另外，这个 demo 里我还特意处理了 [某个边界情况]，代码结构也做了模块化的拆分。
>
> 方便的话，我可以共享屏幕跑一下，或者把 GitHub 链接发给您。”

**场景二：确实没有准备任何 demo（实话实说，但要转化劣势）**

完全没 demo 不一定是大问题，关键是你如何把话题引到你**擅长的事情**上。

> **回答话术：**
> “坦白说，我这次没有准备一个独立的 demo 文件。我之前的实践主要是基于真实业务项目来积累经验的，没有特意把某个功能抽成独立的展示 demo。
>
> 不过，如果您想看我的代码风格或解决问题的思路，我可以直接打开我最近在写的一个业务模块，或者现场用 CodePen/JSFiddle 给您写一个简单的 **[防抖节流 / 数组去重 / 组件通信]** 的小例子。
>
> 比如现在，我可以花两分钟写一个 `useDebounce` 的自定义 Hook，演示一下它的效果。您觉得可以吗？”

**关键点：** 承认没有准备 → 但立刻提供**替代方案**（现场写 / 看业务代码 / 伪代码讲逻辑）。面试官更看重的是你能不能在“没有准备”的情况下还能自信地展示能力。
