## 介绍

权限模型：

- `RBAC`：**常用**。基于角色控制，比如管理员、运营、访客
  - 是 `Role-Based Access Control` 的缩写
- `ABAC`：基于属性控制，比如部门、地区、数据归属、时间段
  - 是 `Attribute-Based Access Control` 的缩写
- `ACL`：基于资源的访问列表，比如某个文档谁可读谁可写
  - 是 `Access Control List` 的缩写

## 中型系统的权限管理方案

中型系统（如企业内部的CRM、OA系统）通常用户量在几百到几千，角色类型相对固定（如管理员、经理、普通员工），权限粒度一般到**菜单/功能按钮**级别。

### 1. 核心模型：RBAC（基于角色的访问控制）

- **用户** → **角色** → **权限**  
  一个用户可以有多个角色，一个角色关联多个权限点（如“查看订单”“编辑商品”）。
- 数据库表设计：`用户表`、`角色表`、`权限表`、`用户-角色关联表`、`角色-权限关联表`。

### 2. 前后端交互流程

**(1) 登录阶段**

- **前端**：提交用户名/密码。
- **后端**：验证身份，生成包含**用户ID、角色列表、权限标识列表**的`Token`（通常是JWT），并返回给前端。

```json
// 后端返回的Token（JWT解码后的载荷示例）
{
  "userId": 123,
  "roles": ["sales", "manager"],
  "perms": ["order:view", "order:edit", "report:view"]
}
```

**(2) 前端权限控制**

- **路由权限**：根据`perms`动态生成可访问的路由菜单。
- **按钮权限**：封装一个自定义指令（如`v-permission="'order:edit'"`），控制按钮显隐。

**(3) 后端权限校验**

- 使用**拦截器或Spring Security**，对每个请求的API进行权限校验：
  - 从请求头中获取Token。
  - 解析出`perms`列表。
  - 判断当前请求的API所需权限（如注解`@PreAuthorize("hasPerm('order:edit')")`）是否在用户的`perms`中。
- **API设计示例**：

  ```txt
  GET /api/orders         → 需要 order:view
  PUT /api/orders/{id}    → 需要 order:edit
  ```

### 3. 适用场景及特点

- 权限变更频率较低（比如新增一个角色/权限点需要管理员手动配置）。
- 前后端约定所有权限标识符（如`order:view`），硬编码在API注解和前端指令中。
- 性能较好：权限校验只是字符串匹配，Token自身包含权限信息，无需查数据库。

## 大型系统的权限管理方案

大型系统（如电商平台、云控制台、企业级中台）用户量可达百万级，角色和权限维度更复杂，需支持**数据权限**（比如只能查看自己部门的订单）、**动态权限分配**、**多租户**等。

### 1. 核心模型：RBAC + 数据规则 + 权限策略引擎

- **基础RBAC** 仍然保留，但权限点可能多达数千个。
- **引入“数据权限”**：通过规则表达式（如`用户.部门ID = 订单.部门ID`）控制行级数据可见性。
- **权限策略引擎**：如使用`OPA（Open Policy Agent）`或自研规则引擎，支持动态编写权限规则（例如“下午6点后禁止导出报表”）。
- **多级组织架构**：权限可按部门、项目、区域等维度继承和隔离。

### 2. 前后端交互流程

**(1) 登录与权限获取**

- 登录后后端返回一个**短时效的Access Token**，但不把所有权限信息塞入Token（Token体积会过大）。
- 前端调用一个**独立的“权限服务”** 异步获取权限树和数据规则：

```json
// 前端请求：GET /api/permissions/my
{
  "menuPerms": ["order:*", "report:view"],        // 功能权限
  "dataRules": {                                   // 数据权限规则
    "order:view": "department_id = {user.dept_id}",
    "order:edit": "creator_id = {user.id}"
  },
  "actionRules": {                                 // 操作约束
    "order:export": "allow_time >= 09:00 && allow_time <= 18:00"
  }
}
```

**(2) 前端控制**

- 使用获取到的`menuPerms`动态构建菜单。
- 按钮级控制同样依赖`menuPerms`。
- **数据权限前端不处理**（只负责展示后端过滤后的数据）。

**(3) 后端权限校验**

- **功能权限**：与中型系统类似，在网关或拦截器中校验API的权限标识。
- **数据权限**：
  - 后端在**服务层（或ORM拦截器层）** 解析`dataRules`，将其动态拼接到SQL的`WHERE`子句中。
  - 例如用户请求`GET /api/orders`时，后端自动加上`WHERE department_id = 当前用户部门ID`。
  - 对于复杂规则（如跨部门访问），可能调用权限规则引擎实时计算。
- **操作约束**：在业务方法执行前校验时间、IP等环境属性。

### 3. 缓存与性能优化

- 权限数据会频繁读取，使用**Redis**存储用户权限的快照，并设置变更事件（当权限修改时，发送消息清空相关用户的缓存）。
- 对于数据权限的SQL改写，可缓存每个用户+资源的查询模板。

### 4. 多服务场景下的交互

- **网关层做统一认证**（验证Token有效性），但不做细粒度权限。
- **业务服务**各自通过拦截器调用权限服务（或本地缓存）进行权限判断。
- 使用**异步消息**同步权限变更（例如用户角色被修改后，通知所有相关服务刷新缓存）。

## 中型和大型系统的前后端交互的核心要点对比

| 维度               | 中型系统                           | 大型系统                                  |
| ------------------ | ---------------------------------- | ----------------------------------------- |
| **权限数据传递**   | Token中携带全部权限标识            | Token只做身份认证，权限单独异步获取       |
| **功能权限校验**   | 后端API注解 + 前端指令匹配字符串   | 类似，但权限点数量多，需分级管理          |
| **数据权限实现**   | 通常不做，或简单写死（如只有本人） | 后端动态SQL改写 + 规则引擎                |
| **权限变更生效**   | Token过期后重新登录生效            | 主动推送清缓存（如WebSocket或短轮询）     |
| **前端状态管理**   | 权限存储于Vuex/Pinia，登录时初始化 | 同样存储，但支持动态刷新（无需重新登录）  |
| **异常处理**       | 无权限时返回403，前端跳转或提示    | 同样403，但可提供“申请权限”按钮与流程联动 |

## 前端详细实现

### 单体应用

#### 一、整体思路：四个权限层级

1. **登录认证**：确定用户是谁（身份校验）。
2. **路由权限**：控制用户能访问哪些页面（如：普通员工无法访问后台管理页）。
3. **菜单权限**：控制侧边栏或导航栏显示哪些入口（根据权限动态生成）。
4. **按钮/元素级权限**：控制页面内具体操作（增、删、改、查按钮）的可见性/可操作性。
5. **页面内数据权限提示**：比如用户点进某页面时，如果没有权限，显示“无权访问”或空状态，而不是直接白屏报错。

#### 二、具体实现步骤

##### 1. 登录与权限数据存储

用户登录后，后端返回**身份凭证（token）**和**权限标识列表**（如路由名、按钮编码）。前端将其存入`localStorage`或状态管理（Vuex/Pinia/Redux）。

```json
// 后端返回示例
{
  "token": "xxx",
  "permissions": [
    "user:add",    // 新增用户按钮权限
    "user:edit",   // 编辑用户按钮权限
    "dashboard:view", // 仪表盘页面权限
    "user:list"       // 用户列表页面权限
  ]
}
```

##### 2. 路由权限（动态路由）

核心：**只挂载用户有权限访问的路由**。

- **静态路由**：登录页、404页、公共页面，在应用初始化时直接挂载。
- **动态路由**：需要权限的页面（如“用户管理”、“系统设置”），在登录成功后，根据后端返回的权限列表**动态添加**到路由表中。

**流程示例（Vue Router）**：

```javascript
// 1. 定义所有需要权限的路由（比如在 router/asyncRoutes.js）
const asyncRoutes = [
  { path: '/user', name: 'User', meta: { permission: 'user:list' }, component: () => import('@/views/User.vue') },
  { path: '/role', name: 'Role', meta: { permission: 'role:list' }, component: () => import('@/views/Role.vue') }
];

// 2. 登录后，根据权限过滤路由
function filterAsyncRoutes(routes, permissions) {
  const res = [];
  routes.forEach(route => {
    if (route.meta?.permission && permissions.includes(route.meta.permission)) {
      res.push(route);
    }
  });
  return res;
}

// 3. 动态添加路由
const accessibleRoutes = filterAsyncRoutes(asyncRoutes, userPermissions);
accessibleRoutes.forEach(route => router.addRoute(route));
```

##### 3. 菜单权限

侧边栏菜单**不应写死**，而是根据用户有权限的路由或后端返回的菜单树，递归生成菜单项。这样无权限的页面入口自然就不会显示。

##### 4. 按钮级权限

封装一个**权限指令**（v-permission）或**函数**，在按钮上使用。

**示例（Vue 自定义指令）**：

```vue
<button v-permission="'user:add'">新增用户</button>
<button v-permission="'user:delete'">删除用户</button>

<script>
// 指令实现
app.directive('permission', {
  mounted(el, binding) {
    const requiredPerm = binding.value;
    const userPermissions = store.state.user.permissions; // 获取当前用户的权限列表
    if (!userPermissions.includes(requiredPerm)) {
      el.parentNode?.removeChild(el); // 无权限则移除按钮
      // 或者 el.style.display = 'none';
    }
  }
});
</script>
```

#### 三、还需要处理的问题

1. **刷新页面时权限丢失**：在路由守卫（`router.beforeEach`）中，检查 store 里是否已有权限，若无则从 token 重新获取权限再动态添加路由。
2. **退出登录**：清除权限数据，并**重置路由**（用 `router.matcher` 重置或重新创建 router 实例）。
3. **403 无权限页**：路由守卫中，若用户试图访问未授权的路由，跳转到 `/403` 或首页。

### 微前端

在大型微前端架构中，权限管理的难点不再是“如何控制按钮”，而是如何确保分布在多个独立应用间的权限能被 **“统一认定、同步更新、精准执行”** 。

微前端架构最大的权限挑战，在于**权限管理和数据的碎片化**。如果没有统一规划，各个子应用各自为政，就会出现同一个用户在应用A里有权限，但在应用B里没有，造成糟糕的体验和安全隐患。因此，构建一套统一、高效的权限同步体系是大型微前端项目成功的关键。

#### 权限管理模式

为解决权限碎片化问题，主流的架构设计是采用**以集中式管理为主，分散式执行为辅**的混合模式。在这种模式下：

- **主应用（Container/Shell）**：作为“大脑”，是唯一的**策略制定者**，负责用户的登录认证、全局面板、所有子应用的接入、全局路由守卫，并维护唯一的“权限真理源”（用户信息、角色、权限标识集合）。
- **子应用（Micro App）**：作为“执行者”，从主应用接收权限数据，主要负责自身业务逻辑和功能界面，并基于接收到的数据进行按钮、页面的精确控制。

#### 权限在应用间的传递方式

明确了职责分工，接下来需要将权限数据从主应用高效地传递给各子应用。以下是几种主流的传递方式：

1. **通过 `props` 传递**：
    - **方式**：在`qiankun`等框架中，主应用可以将用户权限对象作为一个属性，直接传给子应用的`mount`生命周期。
    - **示例**：`registerMicroApps([{ name: 'sub-app', props: { permissions: userPermissions } }])`。

2. **通过全局状态共享**：
    - **方式**：利用`qiankun`内置的`initGlobalState`，可以创建一个全局状态，用户信息更新时，所有已加载的子应用都能实时收到通知并同步更新，保障了状态的一致性。
    - **示例**：`const { onGlobalStateChange, setGlobalState } = initGlobalState(initialState);`。子应用可通过`onGlobalStateChange`实时监听变化。

3. **自定义通信**：对于qiankun等框架，可在主应用中将权限校验工具封装，并通过`props`传递给子应用；子应用可调用主应用提供的`checkPermi`等方法来校验权限。

#### 多层次权限控制方案

##### 1. 应用/路由级权限

- **主应用控制入口**：主应用作为“路由网关”，根据用户权限动态生成可访问的菜单。用户无权访问的子应用，其入口菜单直接被隐藏。
- **主应用路由守卫**：主应用利用全局路由守卫，在用户路由切换时进行拦截，校验目标路由对应的子应用或页面是否在用户的权限列表中。
- **子应用动态路由**：子应用可以根据主应用传入的权限信息，在内部动态注册自己的路由。

##### 2. 页面/功能级权限

权限传递到子应用后，需要进行本地校验。建议在子应用内**封装一个独立的权限工具包或模块**，实现以下复用：

- **工具函数（如`hasPermission`）**：封装一个函数，接收权限码，返回布尔值。
- **自定义指令（如`v-permission`）**：对按钮等元素封装成自定义指令，应用时无需编写额外JS代码，使用便捷。
- **权限组件（如`<Auth />`）**：封装成组件，包裹需要权限的元素，在不同框架的子应用中均可复用。
- **子应用也可独立调用接口**：允许子应用在必要时**独立调用**后端的权限接口，获取自身的权限数据。这确保了子应用在脱离主环境独立运行时，其权限功能依然完整可用。

### iframe

如果微前端采用最朴素的 **iframe** 来实现，权限管理的思路会与 qiankun 等框架有较大不同。iframe 提供了极强的**原生隔离**：独立的上下文、独立的 cookie/localStorage、独立的 origin（如果跨域）。这种隔离既是优点（安全），也是挑战（通信复杂）。

核心原则：**主应用负责认证与权限策略分发，每个 iframe 子应用自己执行权限判定**。

#### 一、整体架构设计

```txt
┌─────────────────────────────────────────────┐
│                 主应用 (Main App)             │
│  - 登录 / SSO                                │
│  - 存储 Token / 权限列表                      │
│  - 渲染菜单、iframe 容器                      │
│  - 通过 postMessage 向 iframe 发送权限/用户信息 │
└─────────────────────────────────────────────┘
        │                          │
        ▼                          ▼
   ┌─────────┐                ┌─────────┐
   │ iframe  │◄──postMessage─►│ iframe  │
   │ (子应用A)│                │ (子应用B)│
   └─────────┘                └─────────┘
   - 监听主应用消息            - 独立的权限校验
   - 动态渲染菜单/按钮         - 调用自己的后端接口时携带 token
```

#### 二、具体实现步骤

##### 1. 主应用：统一登录与 Token 管理

- 用户访问主应用，跳转登录页，输入账号密码。
- 后端返回 **access_token + refresh_token**，以及**权限标识列表**。
- 主应用将 token 存储在 **localStorage**（或 cookie，但需注意 sameSite 问题）。
- 主应用**不关心** iframe 内部的具体业务，只负责分发基础信息。

##### 2. 将 Token 传递给 iframe 子应用

由于 iframe 与主应用可能**同域**或**跨域**，推荐使用 `postMessage` 进行安全通信。

###### 主应用发送消息（iframe 加载完成后）

```javascript
// 在主应用中，获取 iframe 元素
const iframe = document.getElementById('app-iframe');
// 等待 iframe 加载完成
iframe.addEventListener('load', () => {
  iframe.contentWindow.postMessage({
    type: 'AUTH_TOKEN',
    payload: {
      token: 'xxx',
      permissions: ['user:add', 'user:edit', ...],
      userInfo: { name: '张三', userId: 123 }
    }
  }, 'https://subapp.example.com'); // 明确指定目标 origin，防止 XSS
});
```

###### 子应用监听消息

```javascript
// 在子应用（iframe 内部）中
window.addEventListener('message', (event) => {
  // 校验来源（主应用的 origin）
  if (event.origin !== 'https://main.example.com') return;
  
  const { type, payload } = event.data;
  if (type === 'AUTH_TOKEN') {
    // 保存到子应用的 localStorage 或内存状态管理
    localStorage.setItem('token', payload.token);
    localStorage.setItem('permissions', JSON.stringify(payload.permissions));
    // 触发子应用重新渲染或更新权限状态
    appStore.setPermissions(payload.permissions);
  }
});
```

##### 3. 子应用内部的权限控制

子应用拿到权限数据后，就可以像普通单页应用一样做路由/菜单/按钮权限控制：

- **路由权限**：动态添加有权限的路由。
- **菜单权限**：根据权限列表生成侧边栏菜单。
- **按钮权限**：封装自定义指令 `v-permission` 或权限组件。

**注意**：子应用调用自己的后端 API 时，需要在请求头中携带 token（通常是从 localStorage 读取 token）。但 token 是主应用下发的，子应用直接使用即可。若子应用与主应用后端是同一套认证中心，则 token 是通用的。

##### 4. 安全增强：同域情况下的简化方案

如果主应用与所有子应用**部署在同一主域名下**（例如 `main.example.com`、`sub1.example.com`、`sub2.example.com`），可以利用 **cookie 共享** + **同域 localStorage** 来实现更简单的方案：

- 登录成功后，将 token 写入 `.example.com` 域下的 cookie（`domain=.example.com`）或 localStorage（同域可访问）。
- 子应用加载时，直接从自己的 localStorage 或 cookie 读取 token，无需 postMessage。
- 但权限列表依然需要主应用分发（因为权限是集中管理的）。可以：
  - 子应用自己调用一个“获取当前用户权限”的接口（携带 cookie / token），后端返回权限。
  - 或者主应用通过 postMessage 发送权限列表（但 cookie 无法跨子域名直接读取 localStorage）。

##### 5. 实时同步权限变化（重要）

当用户权限发生变更（如管理员修改角色）或退出登录时，所有已打开的 iframe 需要同步更新权限。

**方案**：主应用使用 `postMessage` 广播一条“权限更新”或“登出”消息。

```javascript
// 主应用广播
function broadcastPermissionChange(newPermissions) {
  const iframes = document.querySelectorAll('iframe');
  iframes.forEach(iframe => {
    iframe.contentWindow?.postMessage({
      type: 'PERMISSION_UPDATE',
      payload: { permissions: newPermissions }
    }, '*'); // 生产环境请指定具体 origin
  });
}
```

子应用监听 `PERMISSION_UPDATE` 消息，更新本地存储的权限，并重新渲染。
